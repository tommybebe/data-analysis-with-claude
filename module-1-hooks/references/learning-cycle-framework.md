## 1. 개요

### 1.1 사이클의 설계 원리

하니스 컴포넌트(스캐폴딩, 스킬/훅, 오케스트레이션)는 **추상적 이론이 아니라 직접 경험한 문제**에서 출발해야 합니다. 이 코스의 모든 모듈(1~4)은 동일한 3단계 구조를 반복합니다:

```
경험(Experience) → 회고(Retrospective) → 생성(Generation)
```

- **경험**: 하니스가 없거나 부분적인 상태에서 에이전트 행동을 직접 관찰합니다.
- **회고**: 관찰된 문제를 체계적 질문 프레임워크로 분석하고 해결 방향을 결정합니다.
- **생성**: 분석 결과를 실제 파일/설정/코드로 구현하고 효과를 검증합니다.

### 1.2 핵심 원칙

> **구조는 고정, 내용은 적응**
>
> 사이클의 3단계 구조는 모든 모듈에서 동일합니다. 변하는 것은 각 단계에서 다루는 **관찰 대상**, **회고 영역**, **생성할 컴포넌트**입니다.

| 단계 | 항상 동일한 것 | 모듈마다 달라지는 것 |
|------|---------------|---------------------|
| Phase 1: 관찰 | "에이전트 실행을 관찰하고 행동 패턴을 기록" | 관찰 대상 작업, 관찰 초점 |
| Phase 2: 수정 | "관찰된 문제를 분석하고 기존 컴포넌트를 수정" | 수정 대상, 질문 목록 |
| Phase 3: 창작 | "새로운 하니스 컴포넌트를 설계·구현·검증" | 생성할 컴포넌트, 검증 기준 |

## 1-A. 교육적 근거 (Pedagogical Rationale)

이 섹션은 "왜 관찰 → 수정 → 창작 사이클을 선택했는가?"에 대한 교육학적 근거를 설명합니다. 강사와 코스 설계자가 참조하며, 수강생도 사이클의 의도를 이해하면 더 효과적으로 활용할 수 있습니다.

### 1-A.1 이론적 기반: 경험학습 사이클과의 연관

3단계 학습 사이클은 David Kolb의 **경험학습 이론(Experiential Learning Theory, 1984)**에서 핵심 구조를 차용합니다.

| Kolb의 경험학습 단계 | 본 코스의 대응 단계 | 주요 차이점 |
|---|---|---|
| 구체적 경험 (Concrete Experience) | Phase 1: 관찰 | 에이전트 행동 관찰에 특화 |
| 반성적 관찰 (Reflective Observation) | Phase 2: 수정 (분석) | 4범주 질문 프레임워크로 구조화 |
| 추상적 개념화 (Abstract Conceptualization) | Phase 2: 수정 (매핑) | 하니스 설계 원칙으로 직접 연결 |
| 능동적 실험 (Active Experimentation) | Phase 3: 창작 | 구현 후 before/after 검증으로 강화 |

Kolb 모델과 달리, 본 코스의 사이클은 **하니스 엔지니어링**이라는 단일 주제에 집중하기 위해 각 단계를 좁히고 구체화했습니다. 특히 Phase 2의 "구조화된 회고"는 자유로운 반성이 아니라 **정해진 4범주 질문 프레임워크**를 통해 에이전트 행동 분석에 특화된 회고를 수행합니다.

### 1-A.2 경력자 학습자를 위한 설계 원칙

이 코스의 대상은 **SQL, dbt, Python 경험이 있는 데이터 분석가**입니다. 경력자 학습자(adult learners with domain expertise)는 다음과 같은 특성을 가집니다:

1. **자기 주도적(self-directed)**: 스스로 문제를 정의하고 해결 방향을 결정할 때 더 높은 동기를 유지합니다.
2. **경험 중심(experience-centered)**: 추상적 개념보다 자신의 업무 경험과 연결되는 구체적 사례에서 의미를 찾습니다.
3. **적용 지향적(application-oriented)**: "이걸 왜 배워야 하는가?"에 대한 답이 먼저 있어야 이론을 수용합니다.
4. **즉각적 적용 필요(immediate relevance)**: 배운 내용을 실무에 바로 쓸 수 있어야 한다는 요구가 강합니다.

이러한 특성에 맞게, 사이클은 다음 원칙으로 설계되었습니다:

- **이론 먼저가 아닌 문제 먼저**: Phase 1에서 직접 에이전트 실패를 경험한 후에야 "왜 스캐폴딩이 필요한가?"라는 질문이 의미를 가집니다.
- **추상 원칙보다 구체적 산출물**: 각 Phase는 관찰 노트, 매핑 표, 실제 파일처럼 검증 가능한 산출물을 생성합니다.
- **수강생이 회고를 주도**: 강사가 답을 주는 것이 아니라 4범주 질문을 통해 수강생 스스로 문제를 발견합니다.

### 1-A.3 "경험 먼저" 접근의 효과: 인지 불일치 활용

Phase 1이 "하니스가 없는 상태에서 에이전트 실행"으로 시작하는 것은 의도적입니다. 이는 **인지 불일치(cognitive dissonance)** 를 유발하여 학습 동기를 높이기 위한 설계입니다.

```
수강생의 기대:  "Claude Code는 내가 원하는 분석을 잘 해줄 것이다"
Phase 1 경험:  에이전트가 raw 테이블에 직접 접근하거나, 비용 체크 없이
               대용량 쿼리를 실행하거나, marimo 파일을 잘못된 경로에 생성한다
인지 불일치:   "왜 이렇게 동작하는가? 어떻게 고칠 수 있는가?"
학습 동기:     ↑ (해결책이 필요하다는 내재적 동기 강화)
```

반대로, "스캐폴딩이란 무엇인가?"라는 개념 설명으로 시작하면 경력자 학습자는 "이미 아는 내용"으로 인식하고 주의를 분산할 가능성이 높습니다. 경험이 먼저 올 때 이론은 **해결책**으로 다가옵니다.

### 1-A.4 구조화된 회고의 필요성: 비구조적 회고의 한계 극복

경험학습 연구(Boud, 1985; Schön, 1983)는 경험 자체보다 **경험에 대한 반성**이 학습을 결정한다고 보여줍니다. 그러나 비구조적 반성은 다음 문제를 초래합니다:

- 표면적 관찰에 머물러 근본 원인 분석에 이르지 못함
- 개인 선호에 따라 어떤 문제는 과대평가되고 어떤 문제는 무시됨
- 다음 단계(구현)로 나아가기 위한 명확한 입력이 만들어지지 않음

Phase 2의 **4범주 질문 프레임워크(행동 관찰 → 위험 평가 → 원인 분석 → 해결 설계)**는 이 한계를 해결합니다. 각 범주는 반성을 점진적으로 심화하며, 마지막 범주("해결 설계")는 Phase 3의 구현으로 직접 연결됩니다.

### 1-A.5 반복 구조의 효과: 스키마 구축과 전이 학습

동일한 3단계 사이클이 모든 모듈(1~4)에서 반복되는 것은 **학습 스키마(schema) 구축**을 위한 의도적 설계입니다.

첫 번째 반복 (모듈 1): 사이클의 형식과 규범 학습
두 번째 반복 (모듈 2): 형식에 익숙해지며 내용에 집중
세 번째 반복 (모듈 3): 스키마가 형성되어 복잡한 오케스트레이션 문제에 적용
네 번째 반복 (모듈 4): 스키마를 실무 상황(새 분석 유형, 팀 워크플로)으로 **전이**

이 전이 능력이 최종 학습 목표입니다. 코스가 끝난 후 수강생이 새로운 분석 유형(예: 리텐션 분석, 매출 분석)을 다룰 때, 동일한 3단계 사이클을 적용하여 **스스로 하니스를 설계하고 개선**할 수 있어야 합니다.

### 1-A.6 비교: 대안 교육 방법과의 차이

이 접근 방식이 다른 일반적인 교육 방법과 어떻게 다른지 비교합니다.

| 교육 방법 | 구조 | 경력자 적합도 | 전이 학습 | 본 코스 선택 이유 |
|---|---|---|---|---|
| **강의 중심 (Lecture-based)** | 개념 설명 → 예제 → 연습 | 낮음 (수동적 수용) | 낮음 | ✗ 채택하지 않음 |
| **프로젝트 기반 (Project-based)** | 큰 프로젝트 → 완성 | 중간 (방향성 부재) | 중간 | ✗ 채택하지 않음 |
| **문제 기반 (Problem-based)** | 문제 제시 → 탐구 → 해결 | 높음 | 높음 | ⚠️ 참고함 (더 자유로움) |
| **경험 기반 (Experiential)** | 경험 → 회고 → 개념화 → 실험 | 높음 | 높음 | ✅ 기반 채택 |
| **본 코스 (Experience-Retro-Generate)** | 에이전트 경험 → 구조화 회고 → 구현 | 높음 | 높음 | ✅ 하니스 엔지니어링에 특화 |

본 코스가 단순한 경험 기반 학습과 다른 점은 **하니스 엔지니어링 도메인에 특화된 회고 구조**입니다. 4범주 질문 프레임워크와 문제-컴포넌트 매핑은 일반적 경험 학습에는 없는, 이 코스만의 설계입니다.

### 1-A.7 자기 점검 체크리스트의 교육적 역할

이 코스는 **캡스톤 프로젝트 없이 모듈별 자기 점검 체크리스트**를 평가 방식으로 사용합니다. 이 선택의 이유:

1. **형성 평가(formative assessment)**: 각 모듈이 끝날 때마다 즉각적인 피드백을 제공하여 다음 모듈 진입 전 격차를 해소합니다.
2. **메타인지(metacognition) 강화**: 체크리스트는 "내가 제대로 이해했는가?"를 스스로 묻는 습관을 형성합니다.
3. **경력자 자율성 존중**: 외부 평가자가 아닌 수강생 스스로가 기준에 도달했는지 판단합니다.
4. **실무 이식성**: 자기 점검 체크리스트는 실무에서도 하니스 설계의 완성 기준으로 바로 활용 가능합니다.

캡스톤 프로젝트를 사용하지 않는 이유는 이 코스의 목표가 "하나의 완성된 프로젝트 제출"이 아니라 **반복 가능한 엔지니어링 프로세스 내재화**이기 때문입니다.

## 2. Phase 1: 관찰 (Observe)

### 2.1 정의

수강생이 Claude Code 에이전트와 함께 실제 데이터 분석 작업을 수행하며, 에이전트의 행동 패턴 — 성공, 실패, 예상치 못한 결과 — 을 직접 관찰하는 단계입니다.

### 2.2 목적

- 에이전트의 현재 능력과 한계를 **체감**합니다 (인지 불일치를 유발하여 학습 동기 강화)
- 이후 회고의 **구체적 근거**를 확보합니다
- 하니스 부재 시 어떤 위험이 발생하는지 **실증적으로 확인**합니다

### 2.3 구조

| 항목 | 설명 |
|------|------|
| **입력(Input)** | • 해당 모듈의 실습 가이드<br>• 스타터 레포의 dbt 모델, 합성 데이터<br>• 이전 모듈에서 생성한 하니스 컴포넌트 (모듈 2 이후) |
| **활동(Activity)** | • Claude Code에 분석 프롬프트를 입력하고 에이전트 행동 관찰<br>• 에이전트 출력의 정확성, 일관성, 안전성 평가<br>• 예상과 다른 행동(누락, 오류, 과도한 자율성) 기록 |
| **산출물(Output)** | • **경험 로그**: 프롬프트와 실제 결과를 기록한 메모<br>• **관찰 노트**: 성공/실패/예상외 행동 분류 목록 |

### 2.4 수행 기준

- 최소 **2회** 이상 에이전트 작업 실행 완료
- 각 실행에서 에이전트의 구체적 행동(실행 명령, 생성 파일, 출력 내용)을 기록
- 성공 행동과 실패 행동을 **모두** 관찰

### 2.5 관찰 노트 작성 방법

관찰 노트는 다음 형식으로 작성합니다:

```markdown
## 경험 [번호] 관찰 노트

### 프롬프트
"[에이전트에게 입력한 실제 프롬프트 전문]"

### 에이전트 행동
1. ✅/❌/⚠️ [관찰 항목 1]: [구체적 행동 기술]
2. ✅/❌/⚠️ [관찰 항목 2]: [구체적 행동 기술]
...

### 핵심 관찰
→ [이 경험에서 얻은 인사이트 1~2문장]
```

범례:
- ✅ 예상대로 동작한 행동
- ❌ 예상과 달리 실패한 행동
- ⚠️ 경우에 따라 다른 행동 (일관성 없음)

### 2.6 모듈별 적용 예시

| 모듈 | 전제 조건 | 에이전트 작업 | 관찰 초점 |
|------|-----------|---------------|-----------|
| **모듈 1** | `AGENTS.md` 미존재 | "FitTrack 1월 DAU를 일별 집계해줘" | dbt 모델 계층 인식 여부, raw 테이블 직접 접근 여부 |
| **모듈 2** | 스킬/훅 미설정 (`.claude/` 없음) | "DAU 분석 이슈를 파싱하여 marimo 노트북 생성해줘" | 비용 체크 수행 여부, dbt test 실행 여부, 노트북 규약 준수 |
| **모듈 3** | 워크플로 YAML 미작성 | 수동으로 `auto-analyze` 라벨 부착 후 7단계 흐름 관찰 | 라벨 전환 성공/실패, 단계 간 산출물 전달 정합성 |
| **모듈 4** | 모듈 1~3 하니스 전체 존재 | 새 분석 이슈("주간 리텐션율 추이")로 전체 파이프라인 실행 | 엔트로피(중복 SQL, 미문서 소스) 발생 여부 |

## 3. Phase 2: 수정 (Modify)

### 3.1 정의

Phase 1에서 관찰한 에이전트 행동을 **체계적인 질문 프레임워크**로 분석하여, 하니스로 해결할 수 있는 구체적 문제를 식별하는 단계입니다.

### 3.2 목적

- 개별 관찰을 **패턴으로 일반화**합니다
- 각 문제에 대해 "어떤 종류의 하니스 컴포넌트가 이를 방지할 수 있는가?"를 판단합니다
- Phase 3의 구현 작업을 위한 **명확한 입력**을 생성합니다

### 3.3 구조

| 항목 | 설명 |
|------|------|
| **입력(Input)** | • Phase 1의 경험 로그 및 관찰 노트<br>• 해당 모듈의 구조화된 회고 질문 목록 |
| **활동(Activity)** | • 회고 질문에 대한 서면 답변 작성<br>• 문제 분류(안전성/일관성/완전성/효율성) 및 우선순위 부여<br>• 문제-컴포넌트 매핑 수행 |
| **산출물(Output)** | • **회고 기록**: 회고 질문별 답변 문서<br>• **문제-컴포넌트 매핑 표**: 식별 문제 ↔ 하니스 컴포넌트 유형 |

### 3.4 회고 질문 프레임워크 (4범주)

모든 모듈에서 공통으로 사용하는 4가지 질문 범주입니다:

#### 범주 1: 행동 관찰 (What happened?)

> "에이전트가 [작업]을 수행했을 때, 예상과 달랐던 행동은 무엇인가?"

- 목적: 문제를 **구체적으로 식별**합니다
- 답변 형식: "에이전트는 A를 해야 했지만 B를 했다"

#### 범주 2: 위험 평가 (What's the risk?)

> "이 행동이 프로덕션 환경에서 자동으로 실행되면 어떤 결과를 초래하는가?"

- 목적: 문제의 **심각도를 판단**하여 우선순위를 정합니다
- 답변 형식: "이 행동이 반복되면 [비용 초과 / 데이터 오류 / 신뢰도 손상] 등의 결과가 발생한다"

#### 범주 3: 원인 분석 (Why did it happen?)

> "에이전트가 이렇게 행동한 근본 원인은 무엇인가?"

세 가지 원인 유형 중 하나로 분류합니다:

| 원인 유형 | 설명 | 해결 컴포넌트 |
|-----------|------|---------------|
| **컨텍스트 부족** | 에이전트가 규칙/스키마/제약을 알지 못함 | 스캐폴딩 (AGENTS.md, 데이터 계약) |
| **정책 부재** | 규칙이 선언되었지만 강제 수단이 없음 | 스킬/훅 (권한 정책, pre-commit 훅) |
| **피드백 부재** | 성공/실패 신호가 에이전트에게 전달되지 않음 | 훅/오케스트레이션 (완료 증거, 에러 처리) |

#### 범주 4: 해결 설계 (How to fix it?)

> "이 문제를 반복적으로 방지하려면 어떤 구조를 레포에 추가해야 하는가?"

- 목적: Phase 3의 **구체적 구현 방향**을 결정합니다
- 답변 형식: "[컴포넌트 유형]으로 [구체적 내용]을 추가하면 [기대 효과]를 얻을 수 있다"

### 3.5 문제-컴포넌트 매핑 표 작성 방법

```markdown
| # | 관찰된 문제 | 원인 유형 | 해결 컴포넌트 | 우선순위 |
|---|-------------|-----------|---------------|---------|
| 1 | dry-run 미수행으로 과도한 BigQuery 비용 발생 위험 | 정책 부재 | bq-cost-guard 훅 | 높음 |
| 2 | marimo 파일이 규약 경로가 아닌 루트에 생성됨 | 컨텍스트 부족 | AGENTS.md 파일 경로 규약 섹션 | 중간 |
| 3 | dbt test 미실행으로 데이터 품질 검증 누락 | 피드백 부재 | dbt-auto-test 훅 | 높음 |
```

### 3.6 수행 기준

- 모든 회고 질문에 대한 **서면 답변 완료** (빈 항목 없음)
- 최소 **1건**의 문제를 하니스 컴포넌트 유형에 매핑
- 매핑된 컴포넌트가 해당 모듈의 학습 목표 범위 내에 포함

### 3.7 모듈별 회고 영역

| 모듈 | 회고 영역 | 핵심 분석 관점 |
|------|-----------|----------------|
| **모듈 1** | 초기화/구조/설정 (3개 영역, 6개 질문) | "에이전트에게 컨텍스트가 부족했던 지점은?" |
| **모듈 2** | 스킬 설계/훅 효과/증거 스키마/실무 적용 (4개 영역, 8개 질문) | "정책 없이 발생한 위반과 피드백 없이 놓친 오류는?" |
| **모듈 3** | 워크플로 설계/CI·CD 통합/E2E 조율/실무 적용 (4개 영역, 8+개 질문) | "자동화 실패 지점과 단계 간 정보 손실 지점은?" |
| **모듈 4** | 코스 전체 회고/실무 적용 계획 (2개 영역, 3개 질문) | "하니스 전체의 ROI와 유지보수 부담은?" |

## 4. Phase 3: 창작 (Create)

### 4.1 정의

Phase 2에서 식별한 문제를 해결하는 구체적인 하니스 컴포넌트(규칙, 스킬, 훅, 워크플로 정의)를 구현하고 검증하는 단계입니다.

### 4.2 목적

- 에이전트의 행동을 개선하는 **실질적 산출물**을 레포지토리에 추가합니다
- 동일 프롬프트 before/after 비교로 **구현 효과를 실증**합니다
- 자기 점검 체크리스트로 산출물 품질을 **기계적으로 검증**합니다

### 4.3 구조

| 항목 | 설명 |
|------|------|
| **입력(Input)** | • Phase 2의 문제-컴포넌트 매핑 표<br>• 해당 모듈의 하니스 컴포넌트 명세<br>• `examples/` 디렉토리의 참조 템플릿 |
| **활동(Activity)** | • 하니스 컴포넌트를 코드/설정 파일로 구현<br>• 동일 프롬프트로 에이전트 재실행하여 행동 변화 확인<br>• 자기 점검 체크리스트로 산출물 품질 검증 |
| **산출물(Output)** | • **하니스 컴포넌트 파일**: 모듈별 기대 산출물<br>• **행동 변화 기록**: before/after 비교 문서<br>• **자기 점검 체크리스트**: 완료 기준 충족 여부 확인 |

### 4.4 before/after 비교 수행 방법

Phase 1과 **동일한 프롬프트**를 재실행하고 다음 표 형식으로 비교합니다:

```markdown
| # | 관찰 항목 | Phase 1 (before) | Phase 3 (after) | 변화 |
|---|-----------|-----------------|-----------------|------|
| 1 | BigQuery dry-run 수행 | ❌ 미수행 | ✅ 자동 수행 | 개선 |
| 2 | marimo 파일 경로 | ❌ 루트에 생성 | ✅ notebooks/ 하위 생성 | 개선 |
| 3 | dbt test 실행 | ❌ 미실행 | ✅ 자동 실행 | 개선 |
```

### 4.5 구현 우선순위 결정 방법

Phase 2의 매핑 표를 기준으로 다음 순서로 구현합니다:

1. **안전성** (비용, 데이터 오염) — 가장 먼저 구현
2. **일관성** (규약 준수, 재현성) — 두 번째
3. **완전성** (누락 단계, 피드백 루프) — 세 번째
4. **효율성** (속도, 자동화 범위) — 마지막

### 4.6 모듈별 주요 산출물

| 모듈 | 하니스 계층 | 주요 산출물 | 코드:설정 비율 |
|------|-------------|-------------|---------------|
| **모듈 1** | 스캐폴딩 | `AGENTS.md`, `sources.yml` 데이터 계약, GitHub Issue 템플릿 | 10:90 |
| **모듈 2** | 스킬/훅 | `.claude/settings.json`, `/analyze` 등 커스텀 슬래시 명령 4개, pre-commit 훅 | 60:40 |
| **모듈 3** | 오케스트레이션 | `.github/workflows/auto-analyze.yml`, 라벨 전환 로직, 에러 처리 | 80:20 |
| **모듈 4** | 통합/개선 | 엔트로피 감지 규칙, `AGENTS.md` v2, 워크플로 개선 기록 | 30:70 |

### 4.7 자기 점검 체크리스트 형식

각 모듈 문서에 다음 형식의 체크리스트가 포함됩니다 (상세 항목은 각 모듈 문서 참조):

```markdown
### 모듈 N 자기 점검 체크리스트

| # | 확인 항목 | 검증 방법 | 기준 |
|---|-----------|-----------|------|
| 1 | [산출물] 파일이 존재하는가? | `ls [경로]` | 파일 존재 |
| 2 | [동작]이 예상대로 수행되는가? | [명령] 실행 후 결과 확인 | [기준값] |
| 3 | before/after 비교에서 개선이 확인되는가? | 동일 프롬프트 재실행 | N건 중 M건 이상 개선 |
```

## 7. 반복 가능성 가이드라인 (Repeatability Guidelines)

이 섹션은 강사가 각 모듈에서 사이클을 일관되게 실행하기 위한 실용적 지침을 제공합니다.

### 7.1 일관성 유지 포인트

사이클의 반복성을 높이기 위해 다음 요소를 모든 모듈에서 동일하게 유지합니다:

1. **관찰 노트 형식**: ✅/❌/⚠️ 범례와 "핵심 관찰" 섹션 고정
2. **회고 기록 형식**: 4범주(행동 관찰, 위험 평가, 원인 분석, 해결 설계) 구조 고정
3. **문제-컴포넌트 매핑 표**: 5개 컬럼(번호, 문제, 원인 유형, 컴포넌트, 우선순위) 고정
4. **before/after 비교 표**: 4개 컬럼(번호, 관찰 항목, before, after, 변화) 고정
5. **`evidence/` 디렉토리**: 모든 산출물 문서를 동일한 위치에 저장

### 7.2 적응 허용 범위

다음 요소는 모듈 맥락에 따라 조정 가능합니다:

- Phase 1의 에이전트 작업 프롬프트 (모듈 목표에 맞게)
- Phase 2의 회고 질문 수와 구체적 내용
- Phase 3의 산출물 파일 목록
- 각 Phase의 목표 소요 시간 (수강생 숙련도에 따라 ±50% 허용)

### 7.3 사이클 가속화/감속화 전략

**경력자 그룹** (SQL/dbt 경험 3년+): Phase 1을 단축하고 Phase 3에 집중
- Phase 1: 관찰 1회 → 핵심 문제 1개 집중 (목표 시간 절반)
- Phase 2: 핵심 범주 2개(위험 평가, 해결 설계) 집중
- Phase 3: 추가 컴포넌트(슬래시 명령 커스터마이징 등) 구현 도전

**초심자 그룹** (에이전트 도구 첫 사용): Phase 1과 Phase 2에 충분한 시간 배분
- Phase 1: 추가 관찰 시나리오 1~2개 실행 (예상치 못한 성공 사례 탐색 포함)
- Phase 2: 모든 회고 질문 완성 후 그룹 토론
- Phase 3: `examples/` 참조 템플릿을 최대한 활용하여 구현 부담 경감

### 7.4 트러블슈팅 — 흔한 막힘 지점

| 막힘 지점 | 발생 Phase | 해결 방법 |
|-----------|-----------|-----------|
| 에이전트가 예상대로 동작하여 관찰할 문제가 없음 | Phase 1 | 더 구체적인 위반 시나리오 제공 (예: "SELECT * 사용 명시적 요청") |
| 회고 질문에 "모르겠음" 답변이 많음 | Phase 2 | 관찰 노트를 다시 읽으며 ❌ 항목 하나씩 집중 분석 |
| 컴포넌트 구현이 예상대로 동작하지 않음 | Phase 3 | `examples/` 디렉토리의 참조 구현과 diff 비교 |
| before/after 비교에서 차이가 미미함 | Phase 3 | 훅 등록 방법 재확인 (`.claude/settings.json` 설정 검토) |
